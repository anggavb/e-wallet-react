## a. Perbedaan antara Linux Kernel dan Distro

### Linux Kernel
**Linux Kernel** adalah inti (core) dari sistem operasi Linux. Kernel bertanggung jawab untuk:
- Mengelola hardware (CPU, RAM, storage, perangkat I/O)
- Manajemen proses dan memori
- Menyediakan abstraksi antara hardware dan software
- Mengelola sistem file dan jaringan

Kernel Linux dikembangkan oleh Linus Torvalds sejak 1991 dan bersifat open-source. Versi kernel dapat dicek dengan perintah:
```bash
uname -r
# Contoh output: 5.15.0-78-generic
```

### Linux Distro (Distribusi)
**Linux Distro** adalah sistem operasi lengkap yang dibangun di atas Linux Kernel, dilengkapi dengan:
- Package manager (apt, yum, pacman, dll.)
- Desktop environment (GNOME, KDE, XFCE, dll.)
- Kumpulan software dan utilitas bawaan
- Konfigurasi dan tampilan yang berbeda-beda

### Perbandingan

| Aspek | Linux Kernel | Linux Distro |
|---|---|---|
| Definisi | Inti OS yang berkomunikasi dengan hardware | OS lengkap berbasis kernel |
| Pengembang | Linus Torvalds & komunitas | Berbagai organisasi/komunitas |
| Contoh | Linux 5.15, Linux 6.1 | Ubuntu, Debian, Fedora, Arch Linux, CentOS |
| Fungsi | Manajemen hardware & sumber daya | Pengalaman pengguna lengkap |
| Ukuran | ~100 MB (source code) | 1–4 GB (ISO installer) |

### Analogi
> Kernel ibarat **mesin mobil**, sedangkan distro ibarat **mobil lengkap** — bodinya, interiornya, warnanya — semuanya berbeda meski mesinnya berasal dari keluarga yang sama.

### Contoh Distro Populer
- **Ubuntu** – Cocok untuk pemula, berbasis Debian
- **Fedora** – Selalu menggunakan teknologi terbaru, didukung Red Hat
- **Arch Linux** – Untuk pengguna berpengalaman, sangat dapat dikustomisasi
- **CentOS/Rocky Linux** – Untuk server enterprise
- **Kali Linux** – Dirancang untuk penetration testing

## b. Linux FHS (Filesystem Hierarchy Standard)

**FHS** adalah standar yang mendefinisikan struktur dan isi direktori pada sistem operasi Linux/Unix. Tujuannya agar direktori konsisten di semua distro Linux.

### Struktur Direktori Utama

```
/
├── bin/        # Binary (perintah dasar sistem)
├── boot/       # File bootloader dan kernel
├── dev/        # File perangkat (device files)
├── etc/        # File konfigurasi sistem
├── home/       # Direktori home user
├── lib/        # Library shared sistem
├── media/      # Mount point media removable
├── mnt/        # Mount point sementara
├── opt/        # Software opsional/pihak ketiga
├── proc/       # Informasi proses (virtual filesystem)
├── root/       # Home direktori user root
├── run/        # Data runtime aplikasi
├── sbin/       # System binary (perintah admin)
├── srv/        # Data service (web, FTP, dll.)
├── sys/        # Informasi hardware (virtual filesystem)
├── tmp/        # File sementara
├── usr/        # User utilities dan aplikasi
└── var/        # Data variabel (log, cache, spool)
```

### Penjelasan dan Contoh Kegunaan

#### `/bin` – Essential User Binaries
Berisi perintah dasar yang dibutuhkan saat sistem boot atau dalam single-user mode.
```bash
ls /bin
# Contoh isi: bash, cat, cp, ls, mkdir, mv, rm, echo
```

#### `/etc` – Configuration Files
Berisi semua file konfigurasi sistem dan aplikasi.
```bash
cat /etc/hostname          # Nama host sistem
cat /etc/passwd            # Informasi user
cat /etc/nginx/nginx.conf  # Konfigurasi Nginx
cat /etc/ssh/sshd_config   # Konfigurasi SSH Server
```

#### `/home` – User Home Directories
Setiap user memiliki direktori pribadi di sini.
```bash
ls /home
# Output: alice  bob  charlie
cd /home/alice             # Direktori milik user alice
```

#### `/var` – Variable Data
Berisi data yang terus berubah seperti log dan cache.
```bash
tail -f /var/log/syslog         # Log sistem real-time
tail -f /var/log/nginx/access.log  # Log akses Nginx
ls /var/cache/apt/              # Cache package APT
```

#### `/tmp` – Temporary Files
File sementara yang dihapus saat sistem restart.
```bash
touch /tmp/file_sementara.txt
# File ini akan hilang setelah reboot
```

#### `/proc` – Process Information (Virtual FS)
Bukan direktori nyata di disk, tapi virtual filesystem yang berisi informasi proses kernel.
```bash
cat /proc/cpuinfo    # Informasi CPU
cat /proc/meminfo    # Informasi memori
cat /proc/uptime     # Waktu sejak sistem menyala
ls /proc/1234/       # Informasi proses dengan PID 1234
```

#### `/dev` – Device Files
Representasi perangkat hardware sebagai file.
```bash
ls /dev/sda*    # Hard disk pertama dan partisinya
ls /dev/tty*    # Terminal devices
# /dev/null     # "Black hole" – membuang output
echo "test" > /dev/null
```

#### `/usr` – User Utilities
Berisi mayoritas aplikasi dan utilitas pengguna.
```bash
ls /usr/bin       # Program user (python3, git, vim, dll.)
ls /usr/lib       # Library aplikasi
ls /usr/share     # Data shared (dokumentasi, icons)
```

## c. Sistem Permission dan Owner pada Linux

Linux menggunakan sistem permission berbasis **User/Group/Other** untuk mengontrol akses file dan direktori.

### Tiga Jenis Permission

| Permission | Simbol | Nilai Oktal | Arti pada File | Arti pada Direktori |
|---|---|---|---|---|
| Read | `r` | 4 | Membaca isi file | Melihat daftar isi direktori |
| Write | `w` | 2 | Mengubah/menulis file | Membuat/menghapus file di dalamnya |
| Execute | `x` | 1 | Menjalankan file | Masuk ke direktori (`cd`) |

### Tiga Kategori Owner

- **User (u)** – Pemilik file
- **Group (g)** – Grup yang memiliki akses
- **Other (o)** – Semua pengguna lainnya

### Membaca Permission

```bash
ls -l /etc/passwd
# Output: -rw-r--r-- 1 root root 1872 Apr 10 12:00 /etc/passwd
#         │└┬┘└┬┘└┬┘
#         │ │  │  └─ Other: r-- (read only)
#         │ │  └──── Group: r-- (read only)
#         │ └─────── User:  rw- (read & write)
#         └───────── Tipe file: - (file biasa), d (direktori), l (symlink)
```

### Mengubah Permission – `chmod`

#### Metode Simbolik
```bash
chmod u+x script.sh        # Tambah execute untuk user
chmod g-w file.txt         # Hapus write untuk group
chmod o+r dokumen.pdf      # Tambah read untuk other
chmod a+x program          # Tambah execute untuk semua (a = all)
chmod u+rwx,g+rx,o-rwx secret.sh  # Kombinasi
```

#### Metode Numerik (Oktal)
```bash
chmod 755 script.sh
# 7 = rwx (user)
# 5 = r-x (group)
# 5 = r-x (other)

chmod 644 index.html
# 6 = rw- (user)
# 4 = r-- (group)
# 4 = r-- (other)

chmod 600 ~/.ssh/id_rsa    # Private key SSH (hanya owner)
chmod 777 /tmp/shared/     # Semua bisa baca/tulis/eksekusi (HINDARI di produksi)
```

### Mengubah Owner – `chown`

```bash
chown alice file.txt               # Ubah user owner ke alice
chown alice:developers file.txt    # Ubah user dan group owner
chown -R www-data:www-data /var/www/html  # Rekursif untuk direktori
```

### Mengubah Group – `chgrp`

```bash
chgrp developers project/
chgrp -R docker /var/run/docker.sock
```

### Special Permissions

| Permission | Simbol | Kegunaan |
|---|---|---|
| SetUID (SUID) | `s` pada user | Program berjalan sebagai owner file, bukan user yang menjalankan |
| SetGID (SGID) | `s` pada group | File baru dalam direktori mewarisi group direktori induk |
| Sticky Bit | `t` | Hanya owner file yang bisa menghapus file di direktori (contoh: `/tmp`) |

```bash
chmod u+s /usr/bin/passwd   # SUID – passwd bisa ubah /etc/shadow
chmod g+s /var/shared/      # SGID – file baru ikut group direktori
chmod +t /tmp               # Sticky bit – proteksi penghapusan
ls -l /tmp
# Output: drwxrwxrwt ... /tmp   ← 't' di akhir
```

## d. Prinsip Enkripsi pada SSH

**SSH (Secure Shell)** adalah protokol jaringan yang memungkinkan komunikasi terenkripsi antara dua komputer. SSH menggunakan beberapa lapisan enkripsi.

### Alur Koneksi SSH

```
Client                          Server
  │                               │
  │──── TCP Handshake (port 22) ──►│
  │◄─── Server Public Key ─────────│
  │                               │
  │──── Key Exchange (Diffie-Hellman) ──►│
  │◄──── Session Key Dibuat ──────│
  │                               │
  │──── Autentikasi (password/key)─►│
  │◄─── Akses Diberikan ───────────│
  │                               │
  │════ Komunikasi Terenkripsi ════│
```

### Tiga Lapisan Enkripsi SSH

#### 1. Symmetric Encryption (Enkripsi Simetris)
Digunakan untuk mengenkripsi **seluruh sesi komunikasi** setelah koneksi terbentuk. Kedua pihak menggunakan **kunci yang sama**.

- Algoritma: AES-256, ChaCha20, 3DES
- Kunci dibuat sementara menggunakan **Diffie-Hellman Key Exchange**
- Kunci diperbarui secara berkala selama sesi

```bash
# Melihat algoritma enkripsi yang digunakan
ssh -v user@server 2>&1 | grep "cipher"
```

#### 2. Asymmetric Encryption (Enkripsi Asimetris)
Digunakan untuk **autentikasi dan pertukaran kunci awal**. Menggunakan sepasang kunci: **Public Key** dan **Private Key**.

```bash
# Membuat pasangan kunci SSH
ssh-keygen -t ed25519 -C "email@example.com"
# Menghasilkan:
# ~/.ssh/id_ed25519      ← Private Key (RAHASIA, jangan dibagikan)
# ~/.ssh/id_ed25519.pub  ← Public Key (boleh disebarkan)

# Menyalin public key ke server
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server
# Atau manual:
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
```

#### 3. Hashing (Integritas Data)
Memastikan data tidak dimodifikasi selama transmisi menggunakan **MAC (Message Authentication Code)**.

- Algoritma: SHA-256, SHA-512, HMAC-SHA2

### Proses Autentikasi dengan Key Pair

```
1. Client mengirim permintaan login + public key identifier
2. Server cek public key di ~/.ssh/authorized_keys
3. Server buat random challenge, enkripsi dengan public key client
4. Client dekripsi dengan private key-nya, kirim bukti ke server
5. Server verifikasi → akses diberikan
```

### Contoh Konfigurasi SSH

```bash
# Koneksi dasar
ssh user@192.168.1.10

# Koneksi dengan port kustom dan private key tertentu
ssh -p 2222 -i ~/.ssh/id_ed25519 user@server.com

# SSH Tunneling (Port Forwarding)
ssh -L 8080:localhost:80 user@server.com   # Local forwarding
ssh -R 9090:localhost:3000 user@server.com # Remote forwarding
```

```bash
# /etc/ssh/sshd_config – Konfigurasi keamanan server SSH
PasswordAuthentication no      # Nonaktifkan login password
PermitRootLogin no             # Larang login sebagai root
AllowUsers alice bob           # Hanya izinkan user tertentu
Port 2222                      # Ubah port default
```

## e. Perbedaan antara HTTP dan HTTPS

### HTTP (HyperText Transfer Protocol)
Protokol komunikasi dasar untuk transfer data di web. Berjalan di **port 80** dan tidak memiliki enkripsi.

### HTTPS (HTTP Secure)
Versi aman dari HTTP yang menggunakan **TLS/SSL** untuk enkripsi. Berjalan di **port 443**.

### Perbandingan

| Aspek | HTTP | HTTPS |
|---|---|---|
| Port | 80 | 443 |
| Enkripsi | ❌ Tidak ada | ✅ TLS/SSL |
| Keamanan | Rentan disadap (Man-in-the-Middle) | Data terenkripsi end-to-end |
| Sertifikat | Tidak diperlukan | Diperlukan (SSL/TLS Certificate) |
| Kecepatan | Sedikit lebih cepat | Sedikit lebih lambat (overhead enkripsi) |
| SEO | Peringkat lebih rendah | Google memprioritaskan HTTPS |
| Indikator browser | 🔓 "Not Secure" | 🔒 Gembok |
| Penggunaan | Situs lokal/development | Semua situs produksi |

### Cara Kerja TLS Handshake pada HTTPS

```
1. Client → Server : "Client Hello" (versi TLS, cipher suites)
2. Server → Client : "Server Hello" + Sertifikat SSL (berisi Public Key)
3. Client          : Verifikasi sertifikat ke Certificate Authority (CA)
4. Client → Server : Pre-master secret (dienkripsi dengan public key server)
5. Kedua pihak     : Generate Session Key dari pre-master secret
6. Komunikasi      : Terenkripsi dengan Session Key (symmetric)
```

### Contoh Konfigurasi HTTPS di Nginx

```nginx
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        root /var/www/html;
    }
}

# Redirect HTTP ke HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}
```

### Mendapatkan Sertifikat Gratis dengan Let's Encrypt

```bash
# Install Certbot
apt install certbot python3-certbot-nginx

# Dapatkan dan install sertifikat otomatis
certbot --nginx -d example.com -d www.example.com

# Auto-renewal
certbot renew --dry-run
```

## f. Docker OCI Compliance Standard

### Apa itu OCI?
**OCI (Open Container Initiative)** adalah organisasi open-source yang didirikan pada 2015 oleh Docker, CoreOS, dan perusahaan teknologi lainnya, di bawah naungan Linux Foundation. Tujuannya adalah membuat **standar terbuka** untuk container agar dapat berjalan di berbagai platform dan runtime.

### Spesifikasi OCI

OCI mendefinisikan tiga spesifikasi utama:

#### 1. OCI Image Specification
Standar format image container yang berisi:
- Layer filesystem (berformat tar)
- Image manifest (metadata)
- Image config (environment, entrypoint, dll.)

```bash
# Melihat manifest OCI sebuah image
docker inspect nginx:latest

# Menyimpan image dalam format OCI
docker save nginx:latest | tar -xv
# Menghasilkan struktur:
# manifest.json
# config.json
# layers/
```

#### 2. OCI Runtime Specification
Mendefinisikan bagaimana container **dijalankan** dari sebuah image. Format konfigurasi: `config.json`.

Runtime yang OCI-compliant:
- **runc** – Runtime default Docker
- **crun** – Alternatif lebih ringan berbasis C
- **kata-containers** – Runtime dengan keamanan VM
- **gVisor (runsc)** – Runtime dengan sandboxing kernel

```bash
# Melihat runtime yang digunakan Docker
docker info | grep "Default Runtime"
# Output: Default Runtime: runc
```

#### 3. OCI Distribution Specification
Standar untuk **mendistribusikan** (push/pull) image melalui registry.

```bash
# Pull image dari registry OCI-compliant
docker pull registry.example.com/myapp:v1.0

# Push image ke registry
docker push registry.example.com/myapp:v1.0
```

### Manfaat OCI Compliance

1. **Interoperabilitas** – Image Docker dapat dijalankan oleh Podman, containerd, atau runtime lain
2. **Vendor-neutral** – Tidak tergantung pada satu vendor
3. **Portabilitas** – Container berjalan konsisten di mana saja
4. **Ekosistem luas** – Kubernetes, OpenShift, dll. semua mendukung OCI

```bash
# Contoh: Image Docker dijalankan dengan Podman (OCI-compliant)
podman run nginx:latest

# Image yang sama dapat dijalankan dengan nerdctl (containerd)
nerdctl run nginx:latest
```

## g. Perbedaan antara Container dan VM (Virtual Machine)

### Virtual Machine (VM)
VM mengemulasi **hardware lengkap** dan menjalankan OS penuh di atasnya menggunakan **Hypervisor**.

### Container
Container berbagi **kernel** sistem operasi host dan mengisolasi proses menggunakan fitur kernel Linux: **namespaces** dan **cgroups**.

### Perbandingan Arsitektur

```
┌──────────────────────────┐    ┌──────────────────────────┐
│      VIRTUAL MACHINE      │    │        CONTAINER          │
├──────────────────────────┤    ├──────────────────────────┤
│  App A  │  App B  │ App C │    │  App A  │  App B  │ App C │
├─────────┼─────────┼───────┤    ├─────────┼─────────┼───────┤
│  Libs   │  Libs   │ Libs  │    │  Libs   │  Libs   │ Libs  │
├─────────┼─────────┼───────┤    ├─────────┴─────────┴───────┤
│ Guest OS│ Guest OS│GuestOS│    │     Container Runtime      │
├─────────┴─────────┴───────┤    ├──────────────────────────┤
│       Hypervisor           │    │       Host OS Kernel       │
├──────────────────────────┤    ├──────────────────────────┤
│        Host OS             │    │        Hardware            │
├──────────────────────────┤    └──────────────────────────┘
│        Hardware            │
└──────────────────────────┘
```

### Perbandingan Detail

| Aspek | Container | Virtual Machine |
|---|---|---|
| Ukuran | MB (ringan) | GB (berat) |
| Waktu startup | Detik | Menit |
| Isolasi | Proses (namespace/cgroup) | Hardware penuh |
| OS | Berbagi kernel host | OS penuh terpisah |
| Portabilitas | Sangat tinggi | Terbatas |
| Overhead | Sangat rendah | Tinggi |
| Keamanan | Isolasi lebih lemah | Isolasi lebih kuat |
| Penggunaan resource | Efisien | Lebih boros |
| Hypervisor | Tidak perlu | VMware, KVM, VirtualBox |
| Contoh | Docker, Podman, LXC | VMware ESXi, VirtualBox, KVM |

### Kapan Menggunakan Mana?

**Gunakan Container ketika:**
- Deployment aplikasi microservices
- CI/CD pipeline
- Development environment yang konsisten
- Scaling horizontal yang cepat

**Gunakan VM ketika:**
- Butuh isolasi penuh antar tenant
- Menjalankan OS berbeda (Windows di host Linux)
- Aplikasi legacy yang membutuhkan kernel tertentu
- Kebutuhan keamanan tinggi (multi-tenant cloud)

## h. Definisi dan Manfaat Image Layer pada Docker

### Apa itu Image Layer?
Docker image tersusun dari **lapisan (layer)** yang bersifat read-only. Setiap instruksi dalam `Dockerfile` menciptakan layer baru. Layer-layer ini bertumpuk membentuk filesystem image secara keseluruhan menggunakan konsep **Union File System (UnionFS)**.

### Visualisasi Layer

```
┌─────────────────────────────┐  ← Container Layer (Read/Write)
├─────────────────────────────┤  ← Layer: COPY app/ /app (RW→RO)
├─────────────────────────────┤  ← Layer: RUN npm install
├─────────────────────────────┤  ← Layer: COPY package.json /app
├─────────────────────────────┤  ← Layer: WORKDIR /app
├─────────────────────────────┤  ← Layer: RUN apt-get install nodejs
└─────────────────────────────┘  ← Base Layer: ubuntu:22.04
```

### Contoh Dockerfile dan Layernya

```dockerfile
FROM ubuntu:22.04          # Layer 1: Base OS (~77MB)
RUN apt-get update && \
    apt-get install -y nodejs npm  # Layer 2: Node.js (~200MB)
WORKDIR /app               # Layer 3: Set working directory (~0B)
COPY package.json .        # Layer 4: Copy package file (~1KB)
RUN npm install            # Layer 5: Install dependencies (~50MB)
COPY . .                   # Layer 6: Copy source code (~10MB)
CMD ["node", "index.js"]   # Layer 7: Metadata only (~0B)
```

```bash
# Melihat layer sebuah image
docker history nginx:latest
# OUTPUT:
# IMAGE         CREATED      CREATED BY                     SIZE
# 3f8a4339aadd  2 weeks ago  CMD ["nginx" "-g" "daemon off"] 0B
# <missing>     2 weeks ago  EXPOSE map[80/tcp:{}]           0B
# <missing>     2 weeks ago  COPY /etc/nginx /etc/nginx      26.4kB
# ...
```

### Manfaat Image Layer

#### 1. Caching – Build Lebih Cepat
Docker **menyimpan cache** setiap layer. Jika layer tidak berubah, Docker menggunakan cache dan tidak perlu membangun ulang.

```bash
# Build pertama: semua layer dibangun dari awal
docker build -t myapp:v1 .
# Output: Step 1/7 : FROM ubuntu:22.04  → Pulling...
#         Step 2/7 : RUN apt-get...     → Running...

# Build kedua (hanya source code berubah, layer 1-5 di-cache):
docker build -t myapp:v2 .
# Output: Step 1/7 : FROM ubuntu:22.04  → Using cache ✓
#         Step 2/7 : RUN apt-get...     → Using cache ✓
#         Step 6/7 : COPY . .           → Building... (hanya ini yang baru)
```

#### 2. Efisiensi Storage – Layer Sharing
Layer yang sama tidak disimpan dua kali di disk.

```bash
# myapp-v1 dan myapp-v2 berbagi layer ubuntu:22.04 dan nodejs
docker images
# REPOSITORY  TAG   SIZE
# myapp       v2    340MB  ← Hanya tambahan ~5MB dari v1
# myapp       v1    335MB

docker system df  # Melihat penggunaan storage Docker
```

#### 3. Efisiensi Transfer – Pull/Push Lebih Cepat
Saat pull image dari registry, hanya layer yang **belum ada** yang didownload.

```bash
docker pull nginx:1.25
# Layer 1: Pull complete (sudah ada di cache lokal) ✓
# Layer 2: Pull complete (baru didownload)          ↓
# Layer 3: Pull complete (baru didownload)          ↓
```

#### 4. Immutability – Keamanan dan Konsistensi
Layer read-only memastikan image tidak berubah secara tidak sengaja. Perubahan hanya terjadi pada **container layer** (writable) yang bersifat sementara.

### Best Practice Penulisan Dockerfile untuk Layer Optimal

```dockerfile
# ❌ BURUK: Setiap RUN membuat layer terpisah
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y nginx
RUN rm -rf /var/lib/apt/lists/*

# ✅ BAIK: Gabungkan perintah terkait dalam satu layer
RUN apt-get update && \
    apt-get install -y curl nginx && \
    rm -rf /var/lib/apt/lists/*

# ✅ BAIK: Urutkan instruksi dari yang jarang berubah ke yang sering berubah
FROM node:18                  # Jarang berubah
WORKDIR /app
COPY package*.json ./         # Berubah jika dependency berubah
RUN npm install               # Di-cache selama package.json sama
COPY . .                      # Sering berubah (source code)
```

## i. Kegunaan Docker Volume dan Network

### Docker Volume

**Volume** adalah mekanisme untuk **menyimpan data secara persisten** di luar container. Data dalam container bersifat sementara — jika container dihapus, datanya ikut hilang. Volume memecahkan masalah ini.

#### Jenis Penyimpanan Data di Docker

| Jenis | Lokasi | Kegunaan |
|---|---|---|
| **Volume** | `/var/lib/docker/volumes/` (dikelola Docker) | Data persisten (rekomendasi) |
| **Bind Mount** | Path bebas di host | Development (mount source code) |
| **tmpfs Mount** | RAM | Data sensitif sementara |

#### Membuat dan Menggunakan Volume

```bash
# Membuat volume
docker volume create mydata

# Melihat daftar volume
docker volume ls

# Inspeksi volume
docker volume inspect mydata

# Menjalankan container dengan volume
docker run -d \
  --name postgres-db \
  -v mydata:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:15

# Bind mount (development)
docker run -d \
  --name webapp \
  -v $(pwd)/src:/app/src \   # Source code di-mount langsung
  -p 3000:3000 \
  node:18

# Menghapus volume (data akan hilang!)
docker volume rm mydata
docker volume prune   # Hapus semua volume tidak terpakai
```

#### Contoh Kasus: Database Persisten

```yaml
# docker-compose.yml
version: '3.8'
services:
  db:
    image: mysql:8.0
    volumes:
      - db_data:/var/lib/mysql       # Data MySQL tersimpan di volume
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # Bind mount
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: myapp

volumes:
  db_data:  # Volume bernama – data tetap ada meski container dihapus
```

### Docker Network

**Docker Network** memungkinkan container berkomunikasi satu sama lain dan dengan dunia luar. Setiap container secara default terhubung ke jaringan bridge.

#### Jenis Network Docker

| Driver | Kegunaan |
|---|---|
| **bridge** | Default. Container di host yang sama bisa saling berkomunikasi |
| **host** | Container berbagi network stack host (tidak ada isolasi) |
| **none** | Container tidak memiliki jaringan |
| **overlay** | Komunikasi antar container di host berbeda (Docker Swarm) |
| **macvlan** | Container mendapat IP di jaringan fisik |

#### Membuat dan Menggunakan Network

```bash
# Membuat network kustom
docker network create mynetwork

# Melihat daftar network
docker network ls

# Inspeksi network
docker network inspect mynetwork

# Menjalankan container dalam network tertentu
docker run -d --name backend --network mynetwork myapp-backend
docker run -d --name frontend --network mynetwork myapp-frontend

# Container dalam network yang sama bisa saling resolve via nama container
# frontend bisa ping backend dengan: ping backend (bukan IP!)
```

#### Contoh Kasus: Aplikasi Multi-Container

```yaml
# docker-compose.yml – Aplikasi Web + Database + Redis
version: '3.8'
services:
  web:
    image: myapp:latest
    ports:
      - "80:3000"
    networks:
      - frontend
      - backend
    depends_on:
      - db
      - redis

  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - backend       # Hanya terhubung ke backend network
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: secret

  redis:
    image: redis:7
    networks:
      - backend

networks:
  frontend:           # Network untuk komunikasi eksternal
  backend:            # Network internal, isolasi dari luar

volumes:
  pgdata:
```

```
Arsitektur Network:
Internet → [Port 80] → web → (backend network) → db
                          └──→ (backend network) → redis
db dan redis TIDAK bisa diakses langsung dari internet
```

#### Isolasi Network – Best Practice Keamanan

```bash
# ❌ BURUK: Semua container di network default (bridge)
docker run -d --name db postgres:15
docker run -d --name app myapp

# ✅ BAIK: Pisahkan network sesuai fungsi
docker network create public-net    # Untuk service yang perlu akses luar
docker network create private-net   # Untuk database, internal services

docker run -d --name db --network private-net postgres:15
docker run -d --name app --network public-net --network private-net myapp
```

## j. Definisi dan Tujuan Web Server dan Reverse-Proxy

### Web Server

**Web Server** adalah software yang menerima **HTTP/HTTPS request** dari client (browser) dan mengembalikan respons berupa konten web (HTML, CSS, JS, gambar, dll.) atau meneruskan request ke aplikasi backend.

#### Web Server Populer

| Web Server | Keunggulan | Penggunaan |
|---|---|---|
| **Nginx** | Ringan, performa tinggi, concurrency besar | Static file, reverse proxy, load balancer |
| **Apache HTTP Server** | Fleksibel, banyak modul | Shared hosting, .htaccess support |
| **Caddy** | Otomatis HTTPS, konfigurasi mudah | Development, site kecil-menengah |
| **LiteSpeed** | Sangat cepat, kompatibel Apache | Hosting komersial |

#### Fungsi Web Server

1. **Melayani static file** (HTML, CSS, JS, gambar)
2. **Mengelola koneksi** HTTP/HTTPS dari client
3. **Logging** akses dan error
4. **Kompresi** konten (gzip, brotli)
5. **Caching** konten statis
6. **SSL/TLS termination**

#### Contoh Konfigurasi Nginx sebagai Web Server

```nginx
# /etc/nginx/sites-available/mywebsite.conf
server {
    listen 80;
    listen 443 ssl;
    server_name www.example.com;

    ssl_certificate /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    root /var/www/html;
    index index.html index.htm;

    # Melayani static file
    location / {
        try_files $uri $uri/ =404;
    }

    # Cache untuk asset statis
    location ~* \.(jpg|jpeg|png|gif|css|js)$ {
        expires 30d;
        add_header Cache-Control "public, no-transform";
    }

    # Gzip compression
    gzip on;
    gzip_types text/css application/javascript image/svg+xml;

    # Logging
    access_log /var/log/nginx/mywebsite.access.log;
    error_log  /var/log/nginx/mywebsite.error.log;
}
```

### Reverse Proxy

**Reverse Proxy** adalah server perantara yang menerima request dari client dan **meneruskannya ke server backend** yang sesuai. Client tidak mengetahui server backend mana yang sebenarnya memproses request-nya.

#### Perbandingan: Forward Proxy vs Reverse Proxy

```
FORWARD PROXY:
Client → [Forward Proxy] → Internet
(Client tahu proxy-nya, digunakan untuk: VPN, bypass geo-block)

REVERSE PROXY:
Internet → [Reverse Proxy] → Backend Server(s)
(Client tidak tahu backend-nya, digunakan untuk: load balancing, keamanan)
```

#### Tujuan dan Manfaat Reverse Proxy

1. **Load Balancing** – Mendistribusikan traffic ke banyak server
2. **SSL Termination** – Menangani enkripsi HTTPS di satu titik
3. **Caching** – Menyimpan respons populer untuk mengurangi beban backend
4. **Keamanan** – Menyembunyikan infrastruktur backend dari publik
5. **Kompresi** – Mengompresi respons sebelum dikirim ke client
6. **Rate Limiting** – Membatasi jumlah request per IP
7. **Single Entry Point** – Satu domain/IP melayani banyak layanan

#### Contoh Konfigurasi Nginx sebagai Reverse Proxy

```nginx
# Reverse proxy ke aplikasi Node.js di port 3000
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_cache_bypass $http_upgrade;
    }
}
```

```nginx
# Load Balancing ke 3 backend server
upstream backend_pool {
    least_conn;                        # Algoritma: koneksi paling sedikit
    server backend1.internal:8080;
    server backend2.internal:8080;
    server backend3.internal:8080;
    keepalive 32;
}

server {
    listen 443 ssl;
    server_name www.example.com;

    location / {
        proxy_pass http://backend_pool;
    }
}
```

#### Contoh Arsitektur dengan Nginx Reverse Proxy + Docker

```yaml
# docker-compose.yml
version: '3.8'
services:
  nginx:
    image: nginx:latest
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./ssl:/etc/nginx/ssl
    depends_on:
      - app1
      - app2

  app1:
    image: myapp:v1
    expose:
      - "3000"    # Tidak expose ke host, hanya ke Docker network

  app2:
    image: myapp:v1
    expose:
      - "3000"
```

```nginx
# nginx.conf untuk docker-compose di atas
upstream apps {
    server app1:3000;
    server app2:3000;
}

server {
    listen 80;
    location / {
        proxy_pass http://apps;
    }
}
```

```
Alur Traffic:
User Browser
    │
    ▼ HTTPS (port 443)
[Nginx Reverse Proxy]
    │         │
    ▼         ▼
[app1:3000] [app2:3000]  ← Tidak bisa diakses langsung dari luar
```
